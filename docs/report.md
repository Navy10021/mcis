# MCIS 프로젝트 분석 및 개선 제안 보고서

## 1) 개요

MCIS(Maritime Conflict Intelligence System)는 AIS 데이터를 기반으로 해상 이상행동 탐지와 조기경보 연구를 수행하기 위한 파이썬 프레임워크다. 현재 블랙시(Black Sea) 케이스(`T0=2022-02-24`)를 중심으로, 데이터 로딩→정제→피처→집계→통계분석/모델링→리포팅의 일련 파이프라인이 잘 정리되어 있다.

본 보고서는 코드/구조/테스트/산출물 관점에서 프로젝트 성숙도를 평가하고, 단기·중기 개선 과제를 우선순위로 제시한다.

## 2) 현재 강점

### 2.1 재현성 중심 구조
- `config/settings.yaml` 중심 실행 구조와 CLI 엔트리포인트(`run_pipeline.py`, `run_analysis.py`, `run_model.py`)가 분리되어 실험 재현성이 높다.
- `outputs/`에 메타데이터·테이블·모델 결과·모델카드가 저장되어 결과 추적이 쉽다.

### 2.2 연구 가드레일 명시
- `ROADMAP.md`에 누수 방지, 시계열 분할 원칙, 합성데이터 해석 제한 등 연구 규율이 문서화되어 있다.
- 단일 사건 데이터셋 한계를 인지하고 “supervised 분류의 제한적 사용”을 명확히 한 점이 적절하다.

### 2.3 테스트 커버리지와 안정성
- 전체 테스트(`pytest -q`) 374개가 통과하며, 주요 모듈(로더/정제/분석/모델/CLI)의 회귀 안정성이 좋다.
- 기능 단위 테스트와 E2E성 CLI 테스트가 공존한다.

## 3) 주요 리스크 및 개선 포인트

### 3.1 저장소 내 `__pycache__` 및 산출물 버저닝 정책 불명확
**문제**
- 패키지 하위에 `__pycache__` 파일들이 포함되어 있으며, `outputs/`와 `data/aggregated` 같은 생성 산출물이 저장소에 함께 존재한다.
- 연구 재현을 위한 예시 아티팩트와 실제 실행 산출물의 경계가 모호해질 수 있다.

**개선 제안**
1. `.gitignore` 재정비: `__pycache__/`, `*.pyc`, 실행 산출물(로그/테이블/모델 json) 기본 제외.
2. 필요한 샘플 산출물은 `outputs/samples/`로 분리, 나머지는 CI 아티팩트로 관리.
3. 대용량 데이터는 Git LFS 또는 외부 스토리지 링크 문서화.

### 3.2 경고(Warning) 관리 부족
**문제**
- 테스트 로그에서 `statsmodels`의 `verbose` 관련 deprecation, `shap.summary_plot`의 RNG FutureWarning이 반복된다.
- 현재는 테스트 통과에 영향이 없지만, 의존성 버전 상승 시 실패 요인이 될 수 있다.

**개선 제안**
1. `statsmodels` 호출부에서 deprecated 인자 제거/대체.
2. SHAP 시각화 호출에 explicit RNG 전달 방식으로 변경(가능 시).
3. `pytest.ini`에 경고 정책(`filterwarnings`)을 정의하고, 치명 경고는 단계적으로 fail 처리.

### 3.3 데이터/모델 버전 추적 메타데이터의 표준화 필요
**문제**
- 결과 파일은 잘 저장되지만, “어떤 코드 커밋/어떤 입력 데이터 해시/어떤 설정”으로 생성됐는지 연결 메타정보 표준이 더 강화될 필요가 있다.

**개선 제안**
1. 모든 CLI 실행 시 다음 필드를 공통 저장:
   - git commit hash
   - config snapshot hash
   - input file hash (raw/panel)
   - 실행 환경(Python/패키지 버전)
2. 결과 JSON/모델카드의 공통 헤더 스키마 정의.
3. `outputs/metadata/index.json` 같은 run registry 추가.

### 3.4 설정 스키마와 실제 실행 인자의 일관성 점검 자동화
**문제**
- 현재도 검증이 존재하나, CLI 옵션 추가 시 schema와 validation이 분리되어 drift가 생길 가능성이 있다.

**개선 제안**
1. config schema 기반 CLI 옵션 자동 생성(또는 역검증 테스트).
2. `tests/test_cli.py`에 “schema 필드 누락/초과” 시나리오를 추가해 계약 위반 조기 감지.
3. 필수 설정 키에 대한 문서 자동 생성(README 동기화).

### 3.5 분석 결과 신뢰성 강화를 위한 통계적 강건성 팩 확장
**문제**
- 이벤트 스터디/ITS/Granger 기본 흐름은 있으나, 연구 발표 단계에서 자주 요구되는 민감도/강건성 리포트 자동화가 더 필요하다.

**개선 제안**
1. Placebo cut-date 다중 반복과 분포 요약 자동화.
2. 다중검정 보정(FDR/Holm) 결과를 표준 출력 테이블에 포함.
3. 추정 윈도우 변화(예: ±14/30/60일) 민감도 매트릭스 생성.

### 3.6 운영 관점(DevEx) 개선 여지
**문제**
- 현재 프로젝트는 연구용으로 잘 구성됐지만, 신규 기여자가 “한 번에 어디까지 실행해야 하는지”를 이해하기까지 시간이 걸릴 수 있다.

**개선 제안**
1. `make` 또는 `just` 기반 명령 래퍼 추가:
   - `make test`, `make pipeline`, `make analysis`, `make model`
2. CI에서 최소 3개 단계 분리:
   - lint/type check
   - unit/integration tests
   - smoke E2E (소형 데이터)
3. 실패 시 디버깅 가이드(FAQ)를 README에 보강.

## 4) 우선순위 액션 플랜

### P0 (즉시, 1~3일)
1. `.gitignore` 및 산출물 관리 정책 확정.
2. 경고 원인 2건(statsmodels/SHAP) 수정.
3. 실행 메타데이터에 commit hash + input hash 추가.

### P1 (단기, 1~2주)
1. CLI-스키마 일관성 테스트 강화.
2. 통계 강건성 자동 리포트(placebo 반복, 다중검정 보정).
3. CI 파이프라인에 smoke E2E 추가.

### P2 (중기, 3~6주)
1. 실험 추적 체계(run registry + 모델카드 표준 필드) 정식화.
2. 성능 최적화(대용량 AIS 처리 시 chunking/메모리 전략 점검).
3. 문서 자동화(설정 스키마→README/Docs 반영).

## 5) 기대 효과

- **재현성 강화**: 결과 파일과 실행 맥락의 1:1 매핑 확보.
- **유지보수성 개선**: 경고·스키마 drift 조기 탐지로 업그레이드 비용 감소.
- **연구 신뢰성 향상**: 강건성·민감도 보고 자동화로 대외 검증 대응력 상승.
- **협업 효율 향상**: 신규 참여자의 진입 장벽 완화 및 실험 반복 속도 향상.

## 6) 결론

MCIS는 이미 “연구용 파이프라인”으로서 구조적 완성도가 높은 편이며, 테스트 안정성도 우수하다. 다음 단계의 핵심은 **(a) 산출물/메타데이터 표준화**, **(b) 경고·의존성 리스크 선제 관리**, **(c) 통계 강건성 자동 리포팅 강화**다. 위 세 축을 우선 적용하면, 논문/보고서 수준의 재현성과 신뢰도를 더 안정적으로 확보할 수 있다.
